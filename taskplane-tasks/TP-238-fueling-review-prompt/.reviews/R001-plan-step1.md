# Plan Review — TP-238 Step 1

## Verdict: REVISE

No Step 1 implementation plan or evidence contract was submitted. `PROMPT.md` describes the desired outcome and `STATUS.md` repeats four unchecked headings, but neither fixes the source route, calculation denominator/coverage, field meanings, or output safety rules for this model-visible nutrition workflow.

A revised plan must explicitly specify:

1. **Review modes, arguments, and minimum read route.** Define the session/activity-ID mode and the date-range mode, including defaults, bounds, precedence when activity and dates are both supplied, and when race name/date context is read rather than guessed. Start from `get_athlete_profile` and athlete-local dates; use `resolve_calendar_dates` for relative dates. Name when each of terse `get_activities`, `get_activity_details`, `get_wellness_data`, `get_training_summary`, and `get_events` is required or optional, and why. `get_activities` already has activity load, durations, and fueling fields, so the plan must not make an unbounded details call for every row. Define pagination: fetch all pages needed before saying a range is complete, otherwise state the covered count/window as partial and do not expose the opaque token.

2. **A closed evidence vocabulary and source-labelled output shape.** Lock the exact meanings in the prompt and pack:
   - `carbs_ingested_g`: upstream athlete-logged carbohydrates consumed *during that activity*, in grams; an absent key is a missing log, while a returned numeric zero must not silently be converted to absent.
   - `carbs_used_g`: an upstream estimate of carbohydrates used/burned during the activity—not intake, not a replacement target, and never a substitute numerator for missing ingested grams.
   - wellness `carbs_g`, `calories_intake`, `protein_g`, and `fat_g`: daily wellness dietary fields, distinct from an activity record and not safely summable/subtractable with it without an upstream linkage.
   - activity `calories_burned` and training load: activity context only, never inputs to a calorie, carbohydrate, or deficit prescription.
   - `custom_fields`: only explicitly requested, upstream athlete-defined fields, labelled by their exact code and treated as semantically unknown unless the athlete supplies meaning. They must not be reclassified as nutrition data.

   Define a compact return layout that separately labels sourced activity evidence, sourced daily-wellness evidence, sourced race/calendar context, calculations, data gaps, and clearly separate non-personalized general guidance. It must count/report missing intake, missing/invalid duration, unavailable/Strava-blocked rows, and pages/rows actually covered rather than implying a complete sample.

3. **Reproducible grams-per-hour rules.** Choose and document one duration basis (`moving_time_seconds`, `elapsed_time_seconds`, or an explicit labelled fallback order); do not switch denominators invisibly. State the exact formula and units: `carbs_ingested_g / (selected_duration_seconds / 3600)`. It may be emitted only for an explicitly returned numeric ingested-carbs value and a positive, valid selected duration; no value is calculated for absent carbs, missing/zero/non-positive duration, unavailable activities, or a nonnumeric custom field. If a range-level rate is allowed, define whether it is the sum of valid logged grams divided by the sum of the same valid-session durations, and require the output to state eligible-session/total-session coverage and exclusions. Do not use `carbs_used_g`, calories, load, wellness daily totals, or an invented target to fill either side of the formula.

4. **Freshness, date, and missing-data behaviour.** Require athlete-local activity and wellness windows, preserve activity timezones, and mark current-day `_meta.as_of` rows as partial. Surface wellness `_meta.stale`, field-semantics/provenance warnings, missing fields, and source availability diagnostics. A missing value is neither zero nor evidence of inadequate fueling; a rate cannot be extrapolated to other sessions. State how the response treats a requested race that is absent from `get_events` (no confirmed event, not a fabricated race context).

5. **Health/product boundaries.** State that the prompt is read-only and never calls write/delete tools, `include_full`, streams, or raw payloads. It must not diagnose, assess an eating disorder, prescribe individual nutrition, calculate or recommend carbohydrate/calorie/sodium/fluid/sweat-rate targets, infer a fuel deficit, claim performance or product effects, invent a food/product library, or turn load/calories into dietary advice. Any general planning material must be visibly educational and conditional, separate from logged facts and calculations; it should ask for a qualified sports dietitian/clinician when the user requests individualized or medical advice.

6. **Step ownership and non-vacuous verification.** The requested `go test ./internal/prompts -run 'FuelingReview'` currently matches no test because `fueling_review_test.go` is deferred to Step 2. Either move focused contract tests into Step 1 (and add that file to its artifacts) or replace the checkpoint with a command that exercises an existing change; state which step owns the prompt function, registry entry, golden fixture, and portable-pack linking test so the same files are not ambiguously created twice. The eventual portable pack also needs `docs/prompts/README.md` added to scope and its pack table updated, or the new pack will not be discoverable.
