# Plan Review — TP-229 Step 3

## Verdict: REVISE

The intended semantics are correct, but the submitted one-sentence plan does not cover the transport and response paths that presently encode the incorrect semantics.

1. **Specify the client transport change and exact-wire tests.** `intervals.WriteSportSettingsParams` / `SportSettingsPace` and `writeSportSettingsBody` currently use the nested `Unit` to emit `pace_units`; there is no `pace_load_type` transport field at all (`internal/intervals/sport_settings.go:13-26,67-76`). The plan must add the required `internal/intervals` implementation and local HTTP contract coverage: `threshold_pace` is the computed m/s value, while separately selected `pace_units` is only the existing valid display preference (or the input-derived fallback) and `pace_load_type` is sent according to the declared preservation/Run/Swim derivation rule. Define treatment of `NONE` and unknown/invalid existing values so a bad display token is not preserved as a write unit. Replace the existing wire assertions that expect `255`/`MINS_KM` as the threshold representation with m/s assertions, and cover preserved versus inferred display/load metadata.

2. **Define the write-response migration, not only the request conversion.** `shapeUpdateSportSettingsResponse` currently takes the m/s `updated.ThresholdPace` (or the planned m/s params fallback) and places it directly into `threshold_pace_seconds_per_*` based on `pace_units` (`internal/tools/update_sport_settings.go:277-308`). Without an explicit replacement this will still return false durations after a correct write. The plan must use `units.ParseUnit` plus `response.PaceSecondsFromMetersPerSecond` for the echo, expose an unambiguous `threshold_pace_meters_per_second` fallback for `NONE`, unknown, or unconvertible display units (rather than `threshold_pace_value`), and carry the selected `pace_units_source` and `pace_load_type` truthfully. Add response assertions for a returned and a params-fallback m/s value (including 3.5714285 m/s → 280 s/km), as well as the non-duration fallback cases.

3. **Make percentage-zone coverage and schema artefacts concrete.** Replace the existing full-mode pace-zone test values of `[480,420]` with a valid increasing percentage example such as `[77.5,100]`, assert the exact values reach the client and echo unchanged, and table-test zero, non-finite, >200, duplicate, and descending pace values while leaving power/HR validation unchanged. The safe-mode test should use a valid pace-zone request and prove both profile and writer clients are untouched. Update the input schema description and add a percentage pace-zone example; it must no longer mention seconds-per-distance. Since that changes the registered schema, include `internal/tools/schema_snapshot/update_sport_settings.json`, `web/data/tools.json`, `web/data/tool_schemas.json`, and the `cmd/gendocs/testdata` goldens in the step (or explicitly regenerate them during verification), then run the stated targeted tests plus `go test ./internal/intervals ./cmd/gendocs`.
