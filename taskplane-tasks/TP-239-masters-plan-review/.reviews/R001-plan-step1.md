# Plan Review — Step 1

## Verdict: REVISE

1. **The Step 1 checklist is not yet an implementable evidence contract.** Before editing `catalog.go` or the portable pack, define the mandatory, visibly separate output sections and their provenance: **observed tool evidence** (tool, athlete-local window, freshness/coverage), **athlete-stated preferences** (availability/duration only), **cautious interpretation**, **insufficient evidence/questions**, and **reviewable proposals**. A proposal must be conditional and explicitly not applied. Age/DOB must not be requested or used as evidence or policy; “masters” is an audience label only.

2. **Hard-session spacing and personal-baseline claims lack a supported route.** `compute_load_balance` is a window aggregate and cannot identify individual hard sessions or their spacing. The plan must state that a session is “hard” only when it is explicitly identified by the athlete or by sufficiently detailed, sourced plan/activity intensity evidence; titles, load, calendar proximity, and age are not substitutes. Otherwise return insufficient evidence rather than calculating a gap in chat. Likewise, identify whether and how the deterministic `compute_baseline` tool is used for individual eligible metrics, with its status/sample/missing-day/freshness metadata retained, and prohibit combining metrics into a readiness/risk score.

3. **Add a tool-specific load-ramp/projection boundary.** `get_fitness_projection` has default `weekly_ramp_pct` and `recovery_week_cadence` assumptions. The plan must prohibit presenting those defaults, or newly invented values, as a masters recommendation or as plan evidence. Use explicitly supplied preferences or verbatim/copyable plan targets only, surface all projection assumptions, and return insufficient evidence when the plan does not supply enough load detail. This keeps the shipped prompt separate from the v2.2 rule engine.

4. **Make insufficient-evidence triggers testable.** Map each required gap to behavior: unavailable/ambiguous plan detail or hard-session classification; absent/invalid zones; short, partial, truncated, or missing historical coverage; and stale/partial/missing wellness or provider-native readiness. Each must name the missing evidence, avoid a comparison/conclusion for that dimension, and ask a focused question. Also preserve TP-235 terminology: availability and requested duration are user-supplied context, not inferred hard constraints or an implied session count.

5. **The Step 1 test command cannot verify this step as ordered.** No `MastersPlanReview` test exists, and the planned focused test file is deferred to Step 2, so `go test ./internal/prompts -run 'MastersPlanReview'` would succeed without running coverage. Add a Step 1 boundary test (and list it as an artifact), or move this command to Step 2. The test must assert the no-age-policy/no-medical/no-score/no-write language, section separation, hard-session fallback, projection boundary, and each insufficient-evidence case.
