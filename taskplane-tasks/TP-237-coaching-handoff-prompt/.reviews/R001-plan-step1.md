# Plan Review — TP-237 Step 1

## Verdict: REVISE

No Step 1 implementation plan or handoff contract was submitted. `PROMPT.md` states the desired outcome and `STATUS.md` repeats four unchecked headings, but neither document makes the source, freshness, privacy, pagination, or file/test decisions needed to implement this model-visible contract safely.

A revised plan must explicitly specify:

1. **The exact Markdown output contract.** Name and order the sections and define what may enter each one. In particular, user-stated goals, constraints, and explicitly accepted decisions must remain separate from Icuvisor-sourced current plan/evidence. “Accepted” must not include an assistant suggestion merely mentioned in chat, and calendar state must not be presented as a user decision. Define a compact evidence row shape that includes the claim, source tool, athlete-local evidence date/window, and returned freshness/as-of marker when one exists.
2. **Date and timestamp semantics.** State how the athlete timezone and current anchor are obtained, when `resolve_calendar_dates` is used, and distinguish a record’s date/window from an `as_of` or provider freshness timestamp. Do not require or invent `fetched_at`: the PRD hides generic debug fetch metadata by default. If a tool exposes no trustworthy timestamp, the handoff must say that rather than manufacture one.
3. **Missing, stale, partial, and paginated behavior.** Lock rules for `_meta.stale`, `_meta.missing_fields`, unavailable/Strava-blocked data, current-day partial data, and unresolved tool failures. If `next_page_token` is present, either fetch the needed pages before claiming completeness or label the evidence partial with its covered window/count; do not paste opaque pagination tokens into the handoff. No missing value may be treated as zero or inferred from chat memory.
4. **Privacy and portability boundaries.** Enumerate the output exclusions—not only API keys, tokens, raw athlete IDs, local/config paths, and secrets, but also raw tool payloads and transport metadata—and state that the athlete manually reviews/copies the Markdown. The workflow must not claim that Claude, ChatGPT, Cursor, or any other client automatically imports or remembers it. Define whether sensitive location/health/free-text details are omitted by default or require explicit user approval.
5. **The read-only retrieval route.** List the exact minimum read tools and when each is called, retain terse defaults, prohibit `include_full`/raw streams unless the contract has a narrowly justified exception, prohibit all write/delete tools, and define the `icuvisor_list_advanced_capabilities` fallback rather than allowing chat-side calculations. Also lock the optional lookback/race-context defaults and upper bounds before Step 2 exposes arguments.
6. **Coherent Step 1 artifacts and verification.** The planned command `go test ./internal/prompts -run 'CoachingHandoff'` is currently vacuous because no matching test exists, while `coaching_handoff_test.go` is deferred to Step 2. Either move focused contract tests into Step 1 and add the test file to this step’s artifacts, or change the checkpoint to an honest package-level validation and explain what it exercises. Clarify that Step 1 creates the prompt function/portable contract while Step 2 only registers it and adds the golden, so the same files are not ambiguously “created” twice.

Cross-step, the plan should add `docs/prompts/README.md` to the allowed/documentation scope and update its pack table; that file is the existing index of every portable client prompt pack, and omitting it would leave `coaching-handoff.md` undiscoverable from the prompt-pack documentation.
