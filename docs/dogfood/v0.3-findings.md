## Environment

- Credential source: `.env-dev` maintained out-of-band by the maintainer.
- Test athlete: redacted prefix recorded only in `STATUS.md`.
- Production-account guard: `.env-dev` athlete was compared against the maintainer's primary `.env` athlete without printing either value; they differ.
- Local binary: `bin/icuvisor` rebuilt from the lane before the dogfood run.
- Client: Codex CLI via MCP stdio, `--ephemeral`, with only env var names passed in MCP config.

### Response-size caveat

Codex's final dogfood transcript did not expose per-tool raw response byte counts or token counts, and raw MCP payloads were intentionally not persisted to avoid committing account data. The findings table records this as `Unavailable` rather than fabricating measurements. Future dogfood should add a redacting MCP proxy or client-side byte counter so KR5-style per-tool byte/token sizes can be captured without storing raw payloads.

## v0.4 entry triage decision

### Launch-blocking before v0.4 entry

These findings block treating the v0.3 write catalog as ready for v0.4 entry because they affect required write paths or PRD-required writable fields:

- #6 — `add_or_update_event` NOTE create refused.
- #7 — `update_wellness` subjective write refused.
- #8 — `update_wellness` missing `spO2` schema field.
- #9 — `create_workout` refused.
- #10 — `create_custom_item` validation rejected the documented INPUT_FIELD case.

### Follow-up / seeded-data requirements, not launch blockers by themselves

- `link_activity_to_event`, `add_activity_message`, `delete_activity`, and `delete_gear` need a seeded disposable activity/gear fixture in the test athlete before they can be safely validated end-to-end.
- `apply_training_plan` needs a seeded disposable training plan/folder before dry-run/apply can be validated end-to-end.
- Per-tool byte/token measurement needs a redacting MCP proxy or client-side byte counter; this is useful for v0.4 token-efficiency work but did not invalidate the v0.3 safety findings.
